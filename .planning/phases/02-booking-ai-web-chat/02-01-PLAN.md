<task type="auto">
  <name>Task 1: Service token auth middleware + appointment lookup API</name>
  <files>src/lib/serviceAuth.ts, src/app/api/appointments/lookup/route.ts</files>
  <action>
Create `src/lib/serviceAuth.ts` with two exports:

1. `verifyServiceToken(request: Request): boolean` - checks `Authorization: Bearer {token}` header against `process.env.INTERNAL_SERVICE_TOKEN`. Returns true if match. The token is a long-lived shared secret (not JWT).

2. `requireServiceOrAdmin(request: Request, shopId?: string): Promise<{ authorized: true; error?: never } | { authorized?: never; error: Response }>` - first tries `verifyServiceToken()`. If that fails, falls back to `requireAdmin(request, shopId)` from `src/lib/auth.ts`. Returns the same error response shape. This allows both voice-service (service token) and admin dashboard (JWT) to call the same endpoints.

Create `src/app/api/appointments/lookup/route.ts` with a GET handler:

- Query params: `shopId` (required), `phone` (required), `status` (optional, defaults to non-cancelled)
- Auth: `requireServiceOrAdmin(request, shopId)` - service token or admin JWT
- Rate limit: `lookup:{ip}` 30 req/min
- Query: Find customer by phone + shopId (`@@unique([phone, shopId])`), then find their upcoming appointments (date >= today, status not CANCELLED) ordered by date ASC, limited to 10
- Include: service (name, duration), staff (name), customer (name, phone, email)
- Return: `{ customer: { id, name, phone, email }, appointments: [...] }` or `{ customer: null, appointments: [] }` if no customer found
- Do NOT expose this as a public endpoint - it contains PII

Also update `src/app/api/appointments/[id]/route.ts` PATCH handler: add a service token check at the top. If `verifyServiceToken(request)` is true, skip the `requireAuth` check and allow the status update (the AI is acting on behalf of the patient). Keep the existing admin/customer auth path as fallback.

Create `src/app/api/internal/shop-context/route.ts` with a GET handler:

- Query params: `shopId` (required)
- Auth: `requireServiceOrAdmin(request, shopId)` from `src/lib/serviceAuth.ts`
- Rate limit: `shop-context:{ip}` 30 req/min
- Query: Single Prisma query with includes:
  ```
  prisma.shop.findUnique({
    where: { id: shopId },
    include: {
      services: { where: { isActive: true }, orderBy: { sortOrder: 'asc' }, select: { id, name, duration, price, description } },
      staff: { where: { isActive: true }, orderBy: { sortOrder: 'asc' }, select: { id, name, role } },
      workingHours: { select: { day, openTime, closeTime, isClosed } },
    },
  })
  ```
- Return shape (this IS the ShopContext type the voice-service will use):
  ```json

    "staff": [...],

  }
  ```
- 404 if shop not found, 403 if subscription not active
- Cache: use `cacheGet`/`cacheSet` from `src/lib/cache.ts` with 2-minute TTL (shop context changes rarely but should refresh reasonably)

This consolidates the 4 parallel fetches from the research's `buildShopContext()` pattern into one API call, reducing latency for session initialization.
